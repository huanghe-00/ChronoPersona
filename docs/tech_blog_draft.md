# 带镣铐的建筑：CRDT + MVCC 构建 AI Agent 记忆大脑

> 副标题：在端侧内存与 Token 配额的极限约束下，如何设计一个不会失忆、不串台、可跨本体移植的长期记忆系统

**作者**: ChronoPersona Team  
**日期**: 2026-05-22  
**状态**: W8 初稿

---

## 1. 问题：为什么现有 RAG 不够？

当前主流 AI Companion（如 Character.AI、Replika）的记忆方案可以概括为：

> 向量数据库 + 长上下文窗口 + 系统提示（人设）

这套方案在单会话、单设备、单人格场景下表现尚可，但一旦面临以下三个挑战，就会系统性崩溃：

### 1.1 多端同步冲突

用户在手机上说"我喜欢川菜"，在车机上说"我最近胃不好，不吃辣了"。传统方案要么"后写覆盖"导致信息丢失，要么粗暴合并导致逻辑矛盾（既喜欢川菜又不能吃辣）。

### 1.2 角色人格串台

同一个用户，在 `therapist`（心理咨询师）人格下暴露深层创伤，在 `companion`（伴侣）人格下分享日常喜好。如果所有记忆共享同一向量空间，咨询师会"知道"用户昨晚吃了什么，伴侣会"知道"用户的医疗诊断——这是灾难性的隐私泄露和人格违和。

### 1.3 记忆幻觉与遗忘

LLM 生成的摘要会丢失条件从句（"如果明天不下雨就去爬山" → "用户喜欢爬山"），高频噪声淹没低频关键信号（"我花生过敏"被"我喜欢吃面"覆盖），临时情绪波动被固化为长期人格特征。

## 2. 核心洞察：记忆是带版本的多人协作

我们从分布式系统和数据库领域借用了两个核心抽象：

### 2.1 CRDT：多端同步的数学保证

**冲突-free Replicated Data Type** 保证：无论用户在多少设备上、以何种顺序写入，最终状态都是确定且一致的。

ChronoPersona 自研了 `LWWMap`（Last-Write-Wins Map）+ `HybridTimestamp`（混合逻辑时钟）：

- **物理时间戳 + 逻辑计数器**：比较时先看物理时间，物理时间相同看逻辑计数器，全序可比。
- **Clock-skew 检测**：当两设备物理时间差 > 500ms，不盲目覆盖，而是**保留双版本**并标记 `CONTRADICTS` 边。
- **Add-wins 语义**：冲突时让"新增信息"胜出，而非简单时间戳大的胜出，更符合对话场景。

### 2.2 MVCC：人格隔离的物理保证

**Multi-Version Concurrency Control** 保证：每个 branch（人格）有独立的存储视图，物理隔离而非逻辑隔离。

- `branch_id` 显式传递：所有数据操作（add/retrieve/navigate）必须指定 `branch_id`，禁止默认全局分支。
- `git checkout` 类比：切换人格 = 切换 branch，L1 会话记忆即时清空，L2/L3 按 branch 隔离。
- 跨 branch 共享：仅通过显式 `merge` 操作，且经过 `IPrivacyFilter` + `IRelevanceFilter` 双重过滤。

## 3. 意图图谱：超越向量检索的语义导航

传统 RAG 的缺陷是"语义相近但意图无关"："我手机没电了"和"我手机丢了"向量相似度极高，但需要的回应完全不同。

我们设计了 **IntentGraph**——基于 8 类语义边和 8 类意图策略的图导航：

| 边类型 | 用途 | 示例 |
|--------|------|------|
| `MENTIONS` | 指代消解 | "那个餐厅" → 具体实体 |
| `CAUSED` | 因果回溯 | "为什么焦虑" → 工作压力 |
| `TEMPORAL_NEXT` | 时序推理 | "方案后来怎样" → 实施记忆 |
| `CONTRADICTS` | 冲突保留 | 手机 vs 车机的偏好矛盾 |
| `SIMILAR_TO` | 平行比较 | "川菜和粤菜哪个好" |

检索流程：
1. **意图解析**：用户输入 → `IntentNode` 分类为 `causal_explore` / `temporal_trace` 等
2. **策略加载**：每个意图对应一组允许的边类型和 `max_hops`
3. **图导航**：PostgreSQL Recursive CTE 执行 BFS，按边权重衰减排序
4. **混合召回**：0.6 × 图谱结果 + 0.4 × 向量检索，融合注入 Prompt

## 4. 带镣铐的优化：MVA 阶段的工程取舍

8 周 MVA 不是做"完美的系统"，而是做"能演示、能解释、能扩展"的骨架。以下是我们在资源极限下的关键取舍：

### 4.1 情感引擎：T0 规则 + LSTM，不用端到端 LLM

- **确定性优先**：关键词匹配对"难过/焦虑"的响应 100% 可预期，LLM 可能因 Prompt 变化漂移。
- **资源效率**：Qwen3.5-9B 本地推理 ~100ms+，T0 规则 <1ms。
- **LSTM 作为 Layer 2**：捕捉"连续 5 轮负面输入"这类规则无法覆盖的时序模式。
- **[RL-PLACEHOLDER]**：PPO/GRPO 策略优化明确推迟到 W8+，避免 MVA 阶段引入不可控的复杂性。

### 4.2 具身智能：Token→Action Bridge，不用 VLA

- **零样本迁移**：同一人格驱动 grid_2d / ros2_mobile / MuJoCo，仅换映射字典。
- **情感调制**：CONCERNED 状态 → 速度降低 50%、音量降低 20%。
- **可审计**：每个动作附带 `reasoning` 字段，追溯"为什么"。

### 4.3 评估框架：A1-A11 对抗测试

我们定义了 11 个对抗场景，覆盖记忆召回、角色隔离、多端冲突、情感一致性等维度。关键不是"分数多高"，而是**每个场景都有自动化断言**，确保任何代码变更都不会悄悄破坏已有能力。

## 5. 已知缺陷与 Beyond MVA

MVA 阶段我们主动识别了 15 项生产级缺陷，其中 5 项已在代码中硬化防御，其余 9 项已文档化为路线图：

| 缺陷 | MVA 防御 | 生产级优化 |
|------|---------|-----------|
| 高频旧记忆压制新记忆 | `access_count` 30 天半衰期衰减 | W-TinyLFU 策略 |
| 条件从句丢失 | `CompressedSummary.source_turn_ids` 保留索引 | NLP 条件感知蒸馏器 |
| 反学习滞后 | `_deprecated_edges` + `status` 字段过滤 | 动态重要性重算 |
| 成本失控 | `CostRecord` 记录 | 硬预算截断（80%/100% 分级降级） |

完整路线图见 `docs/beyond_mva.md`。

## 6. 结语

ChronoPersona 的核心理念是：**记忆的可靠性不是锦上添花，而是 AI Companion 的根基**。没有 CRDT 的多端一致性，用户不敢在多设备上使用；没有 MVCC 的人格隔离，用户不敢暴露真实自我；没有意图图谱的语义导航，Agent 永远"感觉健忘"。

这些能力不需要万亿参数模型，而是需要**带镣铐的架构设计**——在端侧内存、Token 配额、开发周期的极限约束下，做出正确的工程取舍。
