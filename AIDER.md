# Aider 开发规范 — ChronoPersona

> 通用编码与协作规范，适用于所有 AI 模型。

---

## 1. 代码风格（强制）
- Python 文件必须使用 `typing` 进行类型注解
- 异常处理必须指定具体异常类型，禁止裸 `except`
- 日志统一使用 `loguru`，关键路径 INFO，错误路径 ERROR
- 注释和 docstring 使用英文，用户交互文本用中文
- 遵循 Google Python Style Guide

## 2. 接口契约（强制）
- 任何新模块必须先定义抽象接口（放在 `contracts/interfaces/`）
- 接口方法必须包含完整 docstring（参数、返回值、可能抛出的异常）
- 所有涉及数据的操作必须显式传递 `branch_id`，禁止使用默认的全局分支

## 3. 原子修改规则
当一个任务涉及多个模块时，必须同时处理以下文件：
1. 接口定义 (`contracts/interfaces/`)
2. 具体实现类
3. Mock 实现 (`mocks/`)
4. 单元测试 (`tests/`)
5. 如果需要，更新 `__init__.py` 的导出

禁止只改实现不改接口、只改接口不更新 Mock、跳过测试。

## 4. [PLACEHOLDER] 处理规则
- 遇到 `[RL-PLACEHOLDER]`、`[VLA-PLACEHOLDER]` 等标记时，**仅保留标记并添加 TODO 注释**，实现一个空函数或 pass 行为，确保编译和测试不受影响。
- **绝对禁止**尝试实现 PPO/GRPO、VLA 微调等标记为 PLACEHOLDER 的复杂算法。

## 5. 测试要求
- 新增接口必须同步生成 `tests/test_xxx.py`，至少包含 3 个测试用例。
- 修改接口必须同步更新所有实现类、Mock 类以及相关的测试。
- Week 1 核心守卫是 `tests/test_mock_pipeline.py`，任何改动都不得破坏其通过性。

### 5.1 测试语义红线（强制）

- **测试断言必须基于业务语义与架构文档**，禁止基于实现内部细节、硬编码巧合数值或临时凑数。
- **当测试失败时，默认假设是"实现缺陷"**，未经架构师审查，禁止直接修改测试数据、预期值或断言逻辑去迁就实现。
- **禁止"测试迁就实现"（test accommodating bug）**：若发现将测试数据从 0.5 改为 1.0 仅为了让断言通过，而业务语义上 0.5 同样是合法输入，则必须回退测试修改并修复实现。
- 任何涉及"修改测试预期值"的提交，必须在 commit message 中标注 `[TEST-REVIEW]`，并在 PR 描述中说明：① 失败根因；② 为什么修改测试而非实现；③ 架构师是否已审查。

### 5.2 测试数据独立性

- 测试用例的输入数据必须覆盖业务边界值（如 NEUTRAL intensity=0.0 / 0.5 / 1.0），不得仅选取实现当前能通过的子集。
- Mock 的确定性行为不得与真实实现的内部算法耦合（如 Mock 返回固定值不应假设真实实现会走相同分支）。


## 6. Git 工作流
- 每个独立任务完成后自动提交，提交信息格式：`[模块] 类型: 简短描述`（如 `[memory] feat: add CRDT sync module`）。
- 若连续 3 次自动修复测试失败，应回退到上次稳定提交（`git reset --soft HEAD~1`），并缩小任务范围。

## 7. 紧急逃生舱
若陷入修改死循环：
1. 使用 `/undo` 回退
2. 使用 `/drop` 清空当前文件上下文
3. 将大任务拆分为「只改一个接口 + 一个实现 + 一个测试」
4. 必要时手动介入
